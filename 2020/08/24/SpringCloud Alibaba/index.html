SpringCloud Alibaba

简介：

SpringCloud Alibaba 是springCloud 下面的一个子项目，是由阿里巴巴和springcloud合作的项目，SpringCloud Alibaba是由Java程序构建其中开源的项目分为5大组件。

下载宝塔

linux可视化界面 命令：yum install -y wget && wget -O install.sh http://download.bt.cn/install/install.sh && sh install.sh

注意：在虚拟机上面 不可以用生成的那个地址访问 必须使用虚拟机的ip来访问

例如：http://192.168.0.184:8888

nginx 在linux中部署

​ 1.下载 nginx tar包 并解压，本文是挂在腾讯云服务器下 放在www/server/nginx目录下；官网下载地址：http://nginx.org/en/download.html

​ 2.进入www/server/nginx/nginx-1.18.0 此目录下有一个默认的配置文件 可以运行 执行命令 ./configure 进行配置， 使用 make 命令来编译， 最后执行 make install 来确认编译

​ 3.使用whereis nginx 命令来找出安装的配置过后的nginx的路径 此例子是在 usr/local/nginx

​ 4.进去到此目录中 核心文件夹 conf 核心配置，这个文件下的nginx.conf文件是主要配置代理的配置 ，sbin 启动类

​ 5.进去 sbin 使用 ./nginx 启动服务，默认端口80 注意放开防火墙

常用命令

​ cd /usr/local/nginx/sbin/

​ ./nginx 启动

​ ./nginx -s stop 停止

​ ./nginx -s quit 安全退出

​ ./nginx -s reload 重新加载配置文件

​ ps aux|grep nginx 查看nginx进程

1.nacos

介绍：

​ nacos 是SpringCloud Alibaba 的核心组件之一。nacos又称注册中心和配置中心。类似于 SpringCloud 中的 Eureck 和 springCloud Config 。nacos最大的不同点就是在 ，它将服务的注册和发现 还有配置中心 集于一身。就不需要像SpringCloud 那样需要两个服务来支撑。。。

nacos 在linux单机

  2.解压后进入到nacos的bin目录中

  3. 使用 ./startup.sh -m standalone  解释：在单机情况下 需要加 -m 参数才能够启动成功。
     4. 进去到logs 目录下使用 cat start.out  查看是否启动成功

3.gateWay网关

简介：

​ 网关：提供客户端请求到网关后，由网关来统一分发请求到具体的微服务，其中网关可以做到一定的限流，权限控制，过滤等功能。

nginx 和 gateWay区别就在于 nginx是相当于一个统一的入口，而gateWay网关就相当于具体到相应的服务。

4.seata 分布式事务

依赖：

​ 在springBoot 单体应用中

​ 在springCloud Alibaba 中

介绍：

​ seata是基于springboot在操作不同的数据库，数据一致问题。

原理：

​ seata 主要有三个重要的角色，分别为： TC (全局事务协调者) 需要一个单独的服务器，TM (全局事务发起则) ，RM (资源管理者) ，seata是一个全局事务来管理RM的所有的子事务。当TM发起全局事务的时候，RM的所有子事务就保持了事务的一致性，如果某个事务抛出异常回滚，那么TM会通知TC将所有的事务全部回滚，这样就实现了全局事务管理。

​ seata 分为三种事务策略，分别为： AT，TCC, SAGA , XA 事务模式 ，目前主流的是：AT > TCC > Saga

​ AT : 表示两阶段提交，第一阶段 ： 获取本地事务，在同一个事务中分别操作原始需要操作的数据和操作回滚表数据。第二阶段提交：如果所有的子事务成功，那么全局事务快速提交，如果子事务中某一个事务失败，那么将原提交的事务数据按照回滚日志(undo_log关系表)做反向补偿，回滚数据。（这也就是说，当最后一个子事务失败，前几个事务提交的数据已经持久化到数据库了，但是可以通过回滚日志做反向操作）；

​ TCC : 它可以跨越关系数据库做提交回滚操作，因为它的两阶段提交需要自己定义，流程基本上跟AT模式相同。但是AT是需要基于关系型数据库。TCC 熟称 “手动版的AT模式”.

部署：

​ 在 linux上部署 seata-server-1.4.0 并通过 sh 脚本运行；表示为TC服务器；

​ 在项目中配置

注意：

​ 在 seata 分布式事务中，全局事务不生效的情况：

1. 当子事务出现异常并没有将事务抛出 而是 捕获了，事务不生效。
   2. 关系数据库不支持事务的时候，全局事务失效。
   3. 当使用AT事务时必须有官方提供的 log 表，做回滚数据，没有则不生效。
   4. 当人为干预了数据库数据，回滚日志跟数据库数据不一致时，全局事务不生效。

当遇到高并发的时候 seata

根据数据库的本地锁 （行锁，表锁） 和 seata 提供的全局锁来保证了一些列的问题。

写隔离 ：

​ 当有多个全局事务同时操作相同的数据的时候，子事务拿到相应的本地事务，将数据修改后，在第二阶段提交的时候必须拿到全局锁才能够正常的提交操作，当一个全局事务拿到全局锁后，其他的事务做重试操作，默认10次重试，如果重试失败，那么回滚数据。

读隔离：、

   seata默认的事务隔离是  “未提交读” 这样会出现脏读数据，那么我们可以修改事务隔离级别，变为已提交读。

安装解压 RocketMq 压缩包

1.启动mqnamesrv 使用命令 nohup sh mqnamesrv &

2.启动mqbroker 使用命令 nohup sh mqbroker -n localhost:9876 &

集群部署：

1. NameService 是无状态的一个服务，每个broke都会在各个NameService上注册自己的服务，所以NameService集群只需要直接启动多个服务即可。
   2.  broke 是整个Mq的核心，消息的发送者会在NameService上找到对应的Broke，然后将消息发送到broke上，broke搭建集群的时候，我们需要详细的配置，在mq有相应的配置模板。一般采用的都是 双主双从 。
   3. 生产者也是无状态的，只需要启动多个服务即可
