springCloud

微服务历史简介

​ 微服务架构基于SOA架构演变过来的

SOA框架缺点：

​ 1.依赖于中心化服务发现机制。

​ 2.因为SOA框架采用SOAP协议（Http+XML），因为XML传输协议比较占用宽带，整个XML报文中有非常多臃肿的数据 ，所以在微服务架构中采用的是 json 以轻量级方式代替XML报文传输。

 3.服务管理非常混乱，缺少服务管理和治理设施不完善。
微服务架构模式

​ 微服务从SOA架构演变过来，比SOA架构上颗粒更精细。让专业的人做专业的事情，目的是为了提高效率。每个服务之间互不影响，每个服务必须独立部署，微服务架构更加体现轻量级。采用restful风格提供API ，也就是Http协议+Json格式，更加轻巧，更加适合于互联网公司敏捷开发，快速迭代产品。

微服务架构与SOA结构区别
1.微服务架构基于SOA架构演变过来，继承了SOA框架的优点，

微服务架构中去除SOA架构中的ESB消息总线，采用Http+json进行传输。

2.微服务架构比SOA架构颗粒更加精细，让专业的人去做专业的事情，目的是提高开发效率，每个服务于服务之间互不影响，微服务架构中，每个服务必须独立部署，微服务架构更加巧，轻量级。

​ 3.SOA架构中可能数据库存储会发生共享，微服务强调每个服务都是单独数据库，保证每个服务于服务之间互不影响。

Eureka注册中心
Eureka用途 ：

​ 主要的作用就是服务的治理，服务的注册

Eureka高可用机制：

​ 原理：eureka服务注册中心如果只有一台，当这个服务注册中心宕机后，那么整个微服务都将无法进行调用，所以我们采用集群的方式来实现服务同步，当一台eureka注册中心宕机后，数据同步到另一台eureka注册中心服务器上。

Eureka集群，同步数据（以两台注册中心为例）

  1. 第一步我们需要将两台注册中心服务器相互注册，以保证当一个注册中心宕机后，另一台注册中心知道，并且及时同步服务。

    eureka服务器1 端口8100

    eureka 服务器2 端口9100

Eureka自我保护机制

原理 ：每个在注册中心注册的服务都会每隔一定的时间像Eureka中发送心跳实例，如果没有心跳实例就会被注册中心剔除该服务，但是在一定时间内大量的服务因为网络的原因，导致服务维持不了心跳，那么注册中心就会自我保护，默认情况下90s的保护时间，在这个时间段Eureka不会剔除服务。

默认 ：

​ 1. eureka默认是开启自我保护机制的，

​ 2.每个服务默认发送心跳实例给注册中心是5s钟发送一次

​ 3.注册中心会每隔30s检索一次服务

使用场景：

​ 上线环境：是需要打开自我保护机制，来保证可用服务因为网络原因不被剔除。

​ 测试环境：一般在测试环境是关闭自我保护机制，因为我们需要频繁的开启服务或关闭服务，自我保护会导致我们注册失败。

代码操作：

​ 第一步，在eureka中的application.yml中关闭自我保护机制

​ 第二步，在每个服务中重新定义心跳实例发送间隔时间，eureka等待下一次心跳的间隔时间规则

Hystrix服务保护框架
服务雪崩

​ 原理 ：tomcat默认情况下只有一个线程池来处理客户端所有的请求，默认线程池中有50个线程来处理请求，如果在一段时间内客户端大量的请求来访问某个服务的一个接口造成请求堆积，当tomcat中50个线程都在处理这个接口请求的时候，那么可能会导致这个服务的别的接口不能都被访问，这样就造成了了服务的雪崩。

连环服务雪崩

​ 原理：当注册中心中第一个服务有某些业务需要调用第二个服务，当第一个服务的某个接口有大量的请求堆积的时候，就会导致第一个个服务雪崩，那么第二个服务也会在处理第一个服务的请求的时候造成雪崩效应，这样的话，就会导致这两个服务另外的接口都不能够被访问，这就形成了连环服务雪崩效应。严重的连环服务雪崩效应会导致整个微服务瘫痪。

Hystrix解决雪崩效应

​ Hystrix 核心功能 ：

​ 1.服务降级

​ 2.服务熔断

​ 3.服务隔离

          4.断路器

​ 服务降级：

​ 原理：当服务达到请求的极限的时候，Hystrix 会执行服务降级，会调用本地方法，来为客户端提示一个友好的提示。例如：当请求堆积的时候，客户端会收到像 “服务器繁忙，请稍后重试” ，“正在排队。。。” 等友好的提示，不让客户端一直等待。

​ 目的：不让客户端一直等待，提高了用户体验。

​ 服务熔断：

​ 原理：当服务请求达阈值时（阈值可以自定义），Hystrix 会自动将服务熔断，拒绝阈值以上的请求，并且调用服务降级给客户端发送一个友好的提示。一般服务熔断和服务降级一起使用。 例如 ：当这个服务请求的阈值是同时可以处理40个请求，但是如果客户端发送100个请求过来，那么会选取开头的20请求，并且实现服务熔断，拒绝剩下的80个请求。

​ 服务隔离：

​ 服务隔离：线程池隔离，信号量隔离

      线程池隔离：

​ 原理：因为默认情况下每个tomcat都只有一个线程池来处理所有的请求，为了解决雪崩效应，可以采用线程池隔离，将一个服务的多个接口按分配不同的线程池，那么每个接口都是不同的线程池来处理，这样就很好的减轻了服务雪崩的问题。

SpringCloud config 配置中心
Config的用途：

​ 当项目上线后，如果我们需要修改配置文件，在传统的模式中，需要重启服务重新加载配置文件。这样的话，就会导致上线的项目必须下线更新。SpringCloud config 配置中心是用来管理配置文件，将配置中的某些内容存在外部服务器（git | svn） ，由程序接口调用来获取外部配置文件内容。

原理：
        1.   将某些配置的内容信息放在 git 或 svn 中持久保存

​ 2. 配置ConfigServer服务器，用来向 git 或 svn 中拉取配置文件内容 

        3.   程序向ConfigServer服务器中，获取文件内容

​ 例如 ：order-impl-service-sit.yml order-impl-service : 为服务名 sit : 为环境 .yml ：文件后缀

SpringCloud Zull 网关

​ 原理：配置网关后，所有的请求都必须先到达网关，再由网关来转发给对应的服务器。

​ 目的：可以隐藏真实的服务地址，提高了服务的安全性。可以配置在网关中配置过滤器来过滤掉某些不合格的请求。

  **在zull 网关中默认开启了 本地负载均衡，所以会有超时时间的问题，所以在yml配置中配置了超时时间**
