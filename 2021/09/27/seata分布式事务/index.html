seata分布式事务

声明：本文只做简单的介绍和本人在搭建已经整合其他组件的时候出现的一些问题，提供了一些相应的解决方案，如果需要详细的介绍，下方附有官方链接，请自行研究。

1.概述

​ 官方文档：http://seata.io/zh-cn/

​ seata-server:1.4.2

​ Seata 是一款开源的分布式事务解决方案，致力于提供高性能和简单易用的分布式事务服务。Seata 将为用户提供了 AT、TCC、SAGA 和 XA 事务模式，为用户打造一站式的分布式解决方案。 在大部分企业中使用 AT 相对够用。

​ AT ：主要是两段式提交基于数据库表的回滚，属于补偿性回滚，对于代码的侵入性不强，但是对于特殊的回滚要求的话没有办法特殊操作。

​ TCC：同样是基于两段式提交，但是是seata提供的接口，我们来实现他所提供的方法，手动编写回滚逻辑，这更加灵活，可以处理特殊的回滚需求，但是对于代码侵入性较强。

2.架构图

seata 有重要的三大角色分别为 TC , TM , RM

seata也是要通过本地事务来进行数据回滚的。

TC : 为这个架构中最核心的服务器，表示全局事务的协调者，全局事务是否提交或回滚。

TM: 全局事务的发起者。相当于事务的发起者。

RM: 本地资源，个人理解就是本地事务。

4.配置介绍

​ seata-server其中最重要的两个配置文件是 file.conf 和 registry.conf

​ file.conf : 主要是配置通过什么方式持久化。默认是 文件 (file) 这样就只能限于某台机器。

​ registry.conf : 主要是配置 seata 后台的配置属性放在哪里，默认也是 file 。

​ 后面统一会将其改为 db 做持久化 , nacos 做配置。

3.避坑

坑一：

​ 1.在我们需要seata管理的每个数据库中必须要有undo_log表，用来存储回滚日志。

坑二：

​ 2.在整合mybatis-plus多数据源的时候 ，采用DS() 注解来选择库，切记不能够在同类中定义不同库的方法并且同方法调用，这样会产生源数据库回滚但分支库不回滚的情况。只能够通过注入的方式再调用就没有问题。

坑三：

​ 如果开启了全局事务，所有的分支事务的本地事务都不能够是默认的事务传播机制(@Transactional(propagation = Propagation.REQUIRED))。

​ 必须是 @Transactional(propagation = Propagation.REQUIRES_NEW) 这种事务传播机制。

​ 原因 ：在不同的库中，都不属于同一个事务，Propagation.REQUIRES_NEW 这个的意思就是说如果有事务就挂起，开启一个新的事务。

坑四：

如果采用seata整合mybatis-plus多数据源配置，需要注意。

1.关闭 seata 的数据源代理，开启mybatis-plus的seata数据源代理。

​ 原因：如果采用seata的数据源代理，seata只能够代理当前的主数据源，代理不到别的数据源，就会导致除了主数据源，其他数据源无法被全局事务管理。

坑五：

​ json序列化的问题

​ seata默认的序列化是 jackson ,官方还提供了fastjson ，keyo的json序列化。

​ 但是如果我们项目中采用 LocalDateTime 的时间类型的话，这几种序列化就都不满足，我们只能够采用自定义的序列化方式，而官方并没有提供相关的接口，所以我们需要采用SPI作为一个扩展类来加载到seata服务服务中，才能够解决这个问题。

1.首先定义自定义的序列化类。

MySeataJacksonUndoLogParser.java

其中 public static final String NAME = “mySeataJackson”; 表示SPI引用的序列化的自定义名字。

2.需要在META-INF中定义一个services包下创建 io.seata.rm.datasource.undo.UndoLogParser 文件名（这个文件名就是seata内部的加载器，我们需要把自定义的序列化的类加载到这个包下的类中），文件中就是自定义序列化类的绝对路径。

以上就是需要注意的点。。。
